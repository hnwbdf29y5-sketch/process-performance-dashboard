# Формулировки для диссертации и защиты

Этот файл фиксирует границы эксперимента, чтобы в тексте работы не было завышенных утверждений.

## Как описывать baseline

Корректная формулировка:

> Для экспериментальной проверки была создана исходная неоптимизированная сборка приложения. В ней обработка локального набора телеметрии технологического процесса выполняется в основном потоке браузера. Такой baseline используется для проверки того, выявляет ли автоматизированная методика превышение performance budget и фиксирует ли эффект после оптимизации.

Не использовать:

> В реальном промышленном приложении случайно были обнаружены все эти проблемы.

Baseline является контролируемым исходным состоянием эксперимента. Его задача — воспроизводимо показать деградацию, которую должен обнаружить контур контроля.

## Как описывать результаты

Корректная формулировка:

> В исходной сборке медианный TBT составил 272 мс для `/dashboard`, 288 мс для `/events`, 305 мс для `/reports` и 296 мс для `/details/line-a`, что превышает порог 200 мс. После переноса обработки телеметрии в Web Worker optimized-сборка показала TBT 0 мс на всех измеряемых маршрутах и Lighthouse Performance Score 100.

Не использовать:

> Оптимизация улучшила все возможные аспекты производительности веб-приложения.

Доказанный результат относится к измеренным лабораторным метрикам: TBT, Lighthouse Performance Score, LCP, CLS, JavaScript transfer, total transfer и request count.

## Как писать про resource budget

Корректная формулировка:

> Пороги LCP, CLS и TBT основаны на общепринятых рекомендациях Core Web Vitals и Lighthouse. Ресурсные пороги для JavaScript, общего объема передачи и количества запросов заданы как проектный performance budget экспериментального прототипа. Они фиксируют допустимый запас роста клиентской части и используются для контроля последующих изменений.

Не использовать:

> Все ресурсные показатели сильно улучшились после оптимизации.

В финальном эксперименте основной эффект связан с TBT. Ресурсные метрики не нарушают budget; на части маршрутов optimized-сборка делает один дополнительный запрос к worker-файлу, что является допустимым обменом ради снятия блокировки UI-thread.

## Как писать про INP и TBT

Корректная формулировка:

> Core Web Vitals включают LCP, INP и CLS. В лабораторных измерениях Lighthouse для диагностики блокировки основного потока используется TBT, который помогает выявлять проблемы JavaScript-нагрузки, потенциально влияющие на интерактивность.

Не использовать:

> В эксперименте измерялся INP.

В текущих отчетах измеряется TBT, а не полевая метрика INP.

## Как описывать CI

Корректная формулировка:

> В GitHub Actions используется два режима. Регулярный workflow контролирует optimized-сборку через строгую конфигурацию Lighthouse CI: если приложение нарушает заданные пороги по Lighthouse Performance, LCP, TBT или CLS, проверка завершается ошибкой. Отдельный ручной workflow выполняет полный эксперимент baseline/optimized/comparison через `npm run perf:experiment`.

Не использовать:

> Все baseline-проблемы блокируют CI.

Baseline нужен для исследовательского сравнения до/после. Регулярный CI контролирует текущее оптимизированное состояние приложения.

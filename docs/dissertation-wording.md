# Формулировки для диссертации и защиты

Этот файл фиксирует границы эксперимента, чтобы в тексте работы не было завышенных утверждений.

## Как описывать baseline

Корректная формулировка:

> Для экспериментальной проверки была создана воспроизводимая исходная версия приложения с контролируемой нагрузкой на основной поток браузера и дополнительными клиентскими ресурсами. Такой baseline-вариант используется для проверки того, выявляет ли автоматизированная методика отклонения от performance budget и фиксирует ли эффект после оптимизации.

Не использовать:

> В реальном приложении случайно были обнаружены все эти проблемы.

Baseline в проекте является исследовательским режимом, а не попыткой выдать искусственную деградацию за случайный дефект.

## Как описывать результаты

Корректная формулировка:

> Эксперимент показал снижение TBT с 270-290 мс до 0 мс, уменьшение JavaScript transfer примерно с 147-148 KB до 65-66 KB, уменьшение общего объема передачи примерно с 171-172 KB до 70-71 KB и снижение количества запросов с 8-9 до 6-7. Lighthouse Performance Score вырос с 88-90 до 100.

Не использовать:

> Оптимизация улучшила все возможные аспекты производительности веб-приложения.

Доказанный результат относится к тем метрикам, которые измерялись в эксперименте: TBT, JavaScript transfer, total transfer, request count и Lighthouse Performance Score.

## Как писать про INP и TBT

Корректная формулировка:

> Core Web Vitals включают LCP, INP и CLS. В лабораторных измерениях Lighthouse для диагностики блокировки основного потока используется TBT, который помогает выявлять проблемы JavaScript-нагрузки, потенциально влияющие на интерактивность.

Не использовать:

> В эксперименте измерялся INP.

В текущих отчетах измеряется TBT, а не полевая метрика INP.

## Как описывать CI

Корректная формулировка:

> В GitHub Actions используется строгая конфигурация Lighthouse CI. Если оптимизированная версия нарушает заданные пороги по Lighthouse Performance, LCP, TBT или CLS, проверка завершается ошибкой. Дополнительно `budget-check.js` проверяет значения по `budgets.json` и формирует Markdown-отчет `reports/strict-budget-check.md`.

Не использовать:

> Все baseline-проблемы блокируют CI.

Baseline нужен для исследовательского сравнения до/после. CI контролирует, что текущая оптимизированная версия соответствует порогам.

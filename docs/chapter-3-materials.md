# Материалы для главы 3

## 3.1 Характеристика экспериментального веб-приложения

В качестве объекта экспериментальной проверки используется frontend-приложение, представляющее собой веб-интерфейс мониторинга параметров технологического процесса. Приложение реализовано с использованием React, TypeScript и Vite и содержит несколько маршрутов: панель мониторинга, журнал событий, отчеты по производительности, страницу пороговых значений и карточку контролируемого объекта.

Назначение приложения состоит в отображении контролируемых параметров технологического процесса: давления линии, температуры реактора, вибрации насоса и энергетической нагрузки. Такой предметный контекст позволяет связать frontend-разработку с направлением автоматизации, поскольку интерфейс рассматривается как часть системы контроля параметров.

Скриншоты русскоязычного интерфейса подготовлены в папке `docs/screenshots/`:

| Файл | Содержание |

| `events.jpg` | журнал событий |
| `reports.jpg` | отчеты по производительности |

## 3.2 Реализация автоматизированного конвейера измерений

Автоматизированный конвейер реализован на основе Lighthouse CI. Конвейер выполняет сборку приложения, запускает preview-сервер, выполняет измерения для заданных маршрутов, сохраняет HTML/JSON-отчеты и формирует сводные таблицы с медианными значениями метрик.

Компоненты конвейера:

| Компонент | Назначение |
| --- | --- |
| `npm run build` | production-сборка frontend-приложения |
| `lighthouserc.cjs` | настройка измерений оптимизированной версии |
| `lighthouserc.baseline.cjs` | настройка измерений исходной версии |
| `budgets.json` | performance budget для ресурсов и метрик |
| `scripts/performance-report.js` | расчет медианных значений по JSON-отчетам |
| `scripts/compare-performance.js` | формирование таблицы сравнения до/после |
| `scripts/budget-check.js` | проверка фактических значений по `budgets.json` |
| `.github/workflows/performance.yml` | запуск проверки в GitHub Actions |

## 3.3 Исходная оценка производительности

Исходная версия измеряется с параметром `?variant=baseline`. В этом режиме приложение включает контролируемую нагрузку на основной поток браузера. Это позволяет воспроизводимо показать проблему, связанную с блокировкой main thread, и затем оценить эффект оптимизации.

Файл с исходными значениями: `reports/baseline-summary.md`.

## 3.4 Примененные методы оптимизации

В оптимизированной версии применены следующие решения:

| Проблема | Метрика | Метод оптимизации | Ожидаемый эффект |
| --- | --- | --- | --- |
| Блокировка основного потока | TBT | устранение искусственной синхронной нагрузки | снижение TBT |
| Рост начального JavaScript | JS transfer, TBT | route-level lazy loading через `React.lazy` и исключение тяжелого baseline-модуля | уменьшение начальной нагрузки |
| Тяжелые графические библиотеки | JS transfer, TBT | легкие SVG-графики вместо chart-библиотеки | снижение объема JS |
| Дополнительный тяжелый ресурс | image transfer, total transfer | исключение тяжелой SVG-схемы из оптимизированной версии | снижение объема передачи |
| Риск сдвигов интерфейса | CLS | фиксированные размеры карточек, графиков и панелей | сохранение визуальной стабильности |
| Нерегулярный ручной аудит | все метрики | Lighthouse CI и performance budget | автоматическое выявление отклонений |

## 3.5 Анализ результатов повторных измерений

Файл с показателями после оптимизации: `reports/performance-summary.md`.

Файл со сравнением исходной и оптимизированной версий: `reports/performance-comparison.md`.

По результатам измерений основной эффект оптимизации связан со снижением Total Blocking Time, JavaScript transfer и общего объема передаваемых ресурсов. В исходной версии TBT составлял 270-290 мс в зависимости от маршрута. После оптимизации TBT снизился до 0 мс на всех измеряемых маршрутах. JavaScript transfer снизился примерно с 147-148 KB до 65-66 KB, а общий объем передачи — примерно с 171-172 KB до 70-71 KB. Интегральная оценка Lighthouse Performance выросла с 88-90 до 100.

## 3.6 Оценка эффективности методики

Экспериментальная проверка показала, что предложенная методика позволяет:

1. автоматически запускать измерения для нескольких маршрутов приложения;
2. фиксировать исходные значения метрик производительности;
3. выявлять отклонения от performance budget;
4. связывать проблемную метрику с конкретным направлением оптимизации;
5. повторно измерять приложение после внесения изменений;
6. формировать таблицы для анализа результатов до и после оптимизации.

Таким образом, разработанный прототип подтверждает применимость автоматизированного контура контроля производительности веб-приложений в процессе frontend-разработки.

## Готовая формулировка вывода по главе 3

В третьей главе реализован экспериментальный frontend-прототип и автоматизированный конвейер измерения производительности. Исходные измерения позволили выявить превышение порогового значения TBT, связанное с блокировкой основного потока браузера, а также дополнительную нагрузку по JavaScript transfer и общему объему переданных ресурсов. После применения оптимизаций и повторного запуска Lighthouse CI значение TBT снизилось до 0 мс, JavaScript transfer уменьшился примерно на 55%, общий объем передачи — примерно на 59%, а интегральная оценка Lighthouse Performance достигла 100 на всех маршрутах. Полученные результаты подтверждают, что предложенная методика позволяет автоматизировать контроль производительности, выявлять проблемные зоны и оценивать эффективность оптимизационных мероприятий.

# Материалы для главы 3

## 3.1 Характеристика экспериментального веб-приложения

В качестве объекта экспериментальной проверки используется клиентское веб-приложение, представляющее собой русскоязычный интерфейс мониторинга параметров технологического процесса. Приложение реализовано с использованием React, TypeScript и Vite.

Основные разделы:

| Раздел | Назначение |
| --- | --- |
| `/dashboard` | панель мониторинга линии А |
| `/events` | журнал событий с фильтрацией и сортировкой |
| `/reports` | сводка измерений и отчетные показатели |
| `/settings` | пороговые значения контроля |
| `/details/line-a` | карточка контролируемого объекта |

Приложение использует локальный детерминированный набор телеметрии за смену: давление линии, температуру реактора, вибрацию насоса, нагрузку и расход. Такой набор позволяет воспроизводимо сравнивать две версии приложения без внешнего API.

## 3.2 Реализация автоматизированного конвейера измерений

Автоматизированный конвейер реализован на основе Lighthouse CI. Он выполняет сборку приложения, запускает preview-сервер, измеряет заданные маршруты, сохраняет HTML/JSON-отчеты и формирует Markdown/JSON/CSV-артефакты.

В проекте используются два режима:

| Режим | Назначение |
| --- | --- |
| `Performance CI` | строгий контроль optimized-сборки при push/PR |
| `Performance Experiment` | ручной полный эксперимент baseline/optimized/comparison |

Компоненты конвейера:

| Компонент | Назначение |
| --- | --- |
| `npm run build:baseline` | сборка исходной неоптимизированной версии |
| `npm run build:optimized` | сборка оптимизированной версии |
| `lighthouserc.baseline.cjs` | 9 прогонов исходной версии |
| `lighthouserc.cjs` | 9 прогонов оптимизированной версии |
| `lighthouserc.strict.cjs` | строгий CI-контроль optimized-сборки |
| `scripts/performance-report.js` | расчет медианных значений |
| `scripts/performance-stats.js` | CSV всех прогонов и min/median/max |
| `scripts/budget-check.js` | проверка значений по `budgets.json` |
| `scripts/compare-performance.js` | сравнение до/после |

## 3.3 Исходная оценка производительности

Исходная версия является отдельной baseline-сборкой. В ней обработка локальной телеметрии выполняется синхронно в основном потоке браузера. Это моделирует типичную проблему клиентских интерфейсов мониторинга: подготовка агрегатов, событий и отчетных данных выполняется во время открытия раздела и блокирует UI-thread.

Файл с исходными медианными значениями: `reports/baseline-summary.md`.

Файл со статистикой всех прогонов: `reports/baseline-stats.md`.

Итоговые baseline-значения:

| Маршрут | Прогонов | Score | LCP, с | TBT, мс | JS, КБ | Передача, КБ |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| `/dashboard` | 9 | 89 | 0.37 | 272 | 67 | 72 |
| `/events` | 9 | 88 | 0.37 | 288 | 66 | 71 |
| `/reports` | 9 | 87 | 0.41 | 305 | 67 | 72 |
| `/settings` | 9 | 100 | 0.36 | 0 | 63 | 67 |
| `/details/line-a` | 9 | 88 | 0.41 | 296 | 68 | 73 |

Таким образом, превышение порога TBT зафиксировано на четырех маршрутах. Легкий раздел `/settings` остается в пределах budget, что подтверждает различие нагрузки между разделами приложения.

## 3.4 Примененные методы оптимизации

Оптимизационные мероприятия выбирались по результатам исходного измерения. Основным нарушением являлось превышение TBT, поэтому приоритет был отдан снижению длительных задач в основном потоке браузера.

| Проблема | Метрика | Метод оптимизации | Ожидаемый эффект |
| --- | --- | --- | --- |
| Обработка телеметрии блокирует UI-thread | TBT | перенос расчета в Web Worker | снижение TBT |
| Открытие разделов запускает тяжелую подготовку данных | TBT, Score | фоновый расчет и обновление интерфейса после результата | рост Performance Score |
| Большой интерфейс не должен загружаться единым блоком | JS transfer | route-level code splitting через `React.lazy` | ограничение начальной JS-нагрузки |
| Риск нерегулярной ручной проверки | все метрики | Lighthouse CI, `budgets.json`, CSV и Markdown-отчеты | воспроизводимый контроль |

Оптимизация не сводится к удалению искусственной задержки. Обе версии используют один и тот же набор телеметрии и один и тот же смысловой расчет. Отличие заключается в месте выполнения тяжелой части: baseline выполняет ее в основном потоке, optimized — через Worker.

## 3.5 Анализ результатов повторных измерений

Файл с optimized-показателями: `reports/performance-summary.md`.

Файл со сравнением до/после: `reports/performance-comparison.md`.

Файл со статистикой всех optimized-прогонов: `reports/optimized-stats.md`.

Итоговое сравнение:

| Маршрут | TBT до, мс | TBT после, мс | Score до | Score после |
| --- | ---: | ---: | ---: | ---: |
| `/dashboard` | 272 | 0 | 89 | 100 |
| `/events` | 288 | 0 | 88 | 100 |
| `/reports` | 305 | 0 | 87 | 100 |
| `/settings` | 0 | 0 | 100 | 100 |
| `/details/line-a` | 296 | 0 | 88 | 100 |

Основной эффект оптимизации выражен в устранении блокировки основного потока. Ресурсные метрики находятся в пределах performance budget. Небольшое увеличение количества запросов на некоторых маршрутах связано с загрузкой worker-файла и является допустимым компромиссом, поскольку ключевое нарушение TBT устранено.

## 3.6 Оценка эффективности методики

Экспериментальная проверка показала, что предложенная методика позволяет:

1. автоматически запускать измерения для нескольких маршрутов приложения;
2. фиксировать исходные значения метрик и все отдельные прогоны;
3. выявлять отклонения от performance budget;
4. связывать проблемную метрику с конкретным направлением оптимизации;
5. повторно измерять приложение после внесения изменений;
6. формировать таблицы для анализа результатов до и после оптимизации.

## Готовая формулировка вывода по главе 3

В третьей главе реализован экспериментальный frontend-прототип и автоматизированный конвейер измерения производительности. Исходные измерения выявили превышение порогового значения TBT на маршрутах `/dashboard`, `/events`, `/reports` и `/details/line-a`. Причина отклонения связана с выполнением обработки телеметрии в основном потоке браузера. После переноса тяжелой обработки в Web Worker повторные измерения показали выполнение performance budget: TBT снизился до 0 мс на всех маршрутах, а Lighthouse Performance Score достиг 100 в optimized-сборке. Полученные результаты подтверждают применимость методики для автоматизированного контроля производительности и оценки эффекта оптимизации клиентской части.

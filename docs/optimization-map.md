| Проблема исходной версии | Метрика, которая фиксирует проблему | Причина | Выполненная оптимизация | Результат |
| --- | --- | --- | --- | --- |
| Блокировка основного потока браузера | TBT | синхронная вычислительная нагрузка в baseline-режиме | исключение блокирующего участка из оптимизированной версии | TBT снизился с 270-290 мс до 0 мс |
| Увеличенный объем JavaScript | JS transfer, total transfer | дополнительный baseline-модуль `heavyBaseline` | модуль загружается только в исходном варианте и отсутствует в optimized-маршрутах | JS transfer снизился примерно с 147-148 KB до 65-66 KB |
| Дополнительный тяжелый графический ресурс | image transfer, total transfer | загрузка SVG-схемы `baseline-process-map.svg` в исходной версии | тяжелая схема исключена из optimized-версии | image transfer снизился с 19 KB до 0 KB |
| Дополнительные сетевые запросы | request count | baseline-версия загружает дополнительные JS/image ресурсы | сокращение набора ресурсов в optimized-версии | количество запросов снизилось с 8-9 до 6-7 |
| Риск нерегулярного ручного контроля | все метрики | ручной аудит не фиксирует регрессии системно | Lighthouse CI, `budgets.json`, `budget-check.js`, GitHub Actions | контроль стал воспроизводимым и автоматизированным |

## Интерпретация для диссертации

Baseline-режим не следует описывать как случайную ошибку реального промышленного интерфейса. Его корректная роль — воспроизводимая исходная версия с контролируемыми деградациями. Такой вариант нужен для проверки методики: система должна обнаружить отклонение, связать его с конкретной метрикой и показать эффект после оптимизации.

Optimized-версия показывает результат устранения контролируемых деградаций: блокирующая нагрузка удалена, тяжелый JS-модуль не загружается, тяжелый SVG-ресурс исключен, а проверка `budgets.json` подтверждает выполнение пороговых значений.
